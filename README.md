# 회계프로그램

> 중소기업을 위한 풀스택 회계 관리 시스템

## 📋 프로젝트 소개

실무에서 사용하던 회계 프로그램의 핵심 기능을 직접 구현한 풀스택 웹 애플리케이션입니다.  
단순한 CRUD를 넘어 실제 업무 프로세스를 반영한 회계 시스템으로, 전표 입력부터 재무제표 작성까지 전 과정을 지원합니다.

### 개발 동기

- 실무에서 사용하던 회계 프로그램의 동작 원리 이해
- 복잡한 비즈니스 로직을 가진 실전 프로젝트 경험
- 풀스택 개발 역량 강화 및 포트폴리오 구축

### 개발 기간

- **기간**: 2025년 11월 ~ 2026년 1월 (약 2개월)
- **작업일지**: 29개
- **개발 방식**: 1인 개발, 매일 작업 및 문서화

---

## 🛠️ 기술 스택

### Backend
- **Runtime**: Node.js 18.x
- **Framework**: Express 4.x
- **Database**: MySQL 8.0
- **Authentication**: JWT (jsonwebtoken)
- **Validation**: express-validator

### Frontend
- **Library**: React 18.x
- **HTTP Client**: Axios
- **Chart**: react-chartjs-2
- **Styling**: CSS3

### Development Tools
- **Version Control**: Git, GitHub
- **Documentation**: Notion (29개 작업일지)
- **Testing**: Postman (API 테스트)



## ✨ 주요 기능

### 1. 인증 및 권한 관리
- JWT 기반 로그인/회원가입
- 회원 타입별 권한 분리 (business/personal)
- 세션 관리 및 자동 로그아웃

### 2. 기초 정보 관리
- **회사 관리**: 사업자등록번호, 개업일, 회계기수 설정
- **직원 관리**: 회사별 직원 등록 및 권한 관리
- **거래처 관리**: 거래처 코드 자동 생성, 검색 기능
- **계정과목**: 계정코드 체계, Generated Column 활용

### 3. 전표 입력
- **일반전표**: 
  - 차변/대변 자동 검증
  - 전표번호 자동 생성
  - 계정코드/거래처코드 검색 기능
  - 인라인 수정 모드
- **매입매출전표**: 
  - 라인별 입력 방식
  - 부가세 자동 계산
  - 합계 자동 검증

### 4. 장부 조회
- **계정별원장**: 계정과목별 거래 내역 및 잔액 조회
- **거래처별원장**: 거래처별 채권/채무 관리
- **기간별 조회**: 월/일 범위 지정, 유연한 검색 조건

### 5. 회계 연도 관리
- **회계기수 설정**: 회사별 회계기간 관리
- **기수별 데이터 분리**: 연도별 독립적인 데이터 관리
- **날짜 자동 검증**: 개업일 이전 전표 입력 방지

### 6. 결산
- **결산전표 자동 생성**: 재고자산 조정
- **손익계산서**: 수익/비용 → 당기순이익 대체
- **이익잉여금처분계산서**: 기수별 처분 내역 관리
- **대차대조표**: 자산/부채/자본 현황
- **합계잔액시산표**: 이월잔액 + 당기발생액

### 7. 마감 및 이월
- **이월 기능**: 자산/부채/자본 계정 다음 기수로 이월
- **마감 기능**: 마감된 기수는 조회만 가능
- **마감 취소**: 필요시 마감 해제 가능

### 8. 대시보드
- **주요 지표**: 매출, 매입, 이익, 현금 현황
- **전월 대비 분석**: 증감률 표시
- **차트 시각화**: react-chartjs-2 활용
- **부가세 신고 일정**: 다음 신고 기한 표시

### 주요 기술적 도전과 해결

#### 1. 회계연도 설계 오류
**문제**: 25개 기능 구현 후 회계연도 필요성 발견  
**해결**: 전체 시스템 수정, 모든 전표와 원장에 회계기수 추가  
**교훈**: 핵심 기능은 초기 설계 단계에서 파악해야 함

#### 2. 2026년 문제
**문제**: 시스템 날짜 기준으로 연도 처리 → 2026년에 2025년 전표가 2026년으로 인식  
**해결**: 회계기수의 startDate에서 연도 추출하도록 수정  
**교훈**: 날짜 처리는 항상 비즈니스 로직 기준으로

#### 3. 컴포넌트 중복
**문제**: 일반전표와 매입매출전표에 동일한 검색 로직 중복  
**해결**: 통합 컴포넌트 생성 → 311줄 코드 감소  
**교훈**: 재사용 가능한 컴포넌트 설계의 중요성

#### 4. 결산전표 삭제 로직
**문제**: "결산"으로 시작하는 모든 전표 삭제 → 수동 입력 전표도 삭제됨  
**해결**: 구체적인 식별자 사용 필요  
**교훈**: 데이터 삭제 조건은 매우 구체적이어야 함
